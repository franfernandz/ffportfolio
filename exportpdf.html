Automating a Manual Document Indexing Workflow into a Self-Service Tool

Replaced a manual document indexing process with a zero-installation browser tool, reducing a 2–3 day task to under 5 minutes and enabling non-technical staff to generate structured datasets on demand.

Impact:

  • Reduced task time from days to minutes
  • Eliminated manual data entry errors
  • Enabled self-service access for non-technical users
  • Delivered end-to-end solution in a single iteration

Project Overview

Led the design and development of a lightweight, browser-based tool to automate document indexing across a fragmented public archive, without requiring changes to existing infrastructure.

The solution transformed a one-off operational request into a reusable internal capability.

Problem

A request was made to manually compile an Excel file containing metadata and download links for 500+ government documents distributed across multiple directories.

The existing process required:

  • Navigating multiple pages
  • Manually identifying documents
  • Copying links and metadata one by one
  • Compiling an Excel file

This created:

  • High time cost (multiple days of work)
  • High risk of human error
  • No reusable system for future updates

Insight

The request was framed as a data delivery task, but the real problem was a repeatable workflow with no tooling.

Any manual solution would:

  • Solve the immediate need
  • Fail to scale for future updates
  • Reintroduce the same cost over time

Strategy

Transform a one-time request into a self-service tool that:

  • Automates document discovery and extraction
  • Requires zero technical setup
  • Outputs data in the required format (Excel)
  • Operates entirely on the client side

Key Decisions & Trade-offs

1- Zero-Installation Architecture (Adoption vs Capability)

Delivered as a single browser-based tool.

  • No installation or setup required
  • Accessible to non-technical users
  • Limited by browser execution constraints

Prioritized usability over technical flexibility

2- Output Compatibility (Speed vs System Evolution)

Maintained Excel as output format.

  • Immediate compatibility with existing workflows
  • No need for stakeholder alignment
  • Does not push toward more scalable data systems

Prioritized delivery speed over long-term data standardization

3- No Backend / Self-Contained (Scope vs Extensibility)

  • No infrastructure dependencies
  • Instant deployment
  • No persistence or historical tracking

Prioritized feasibility within constraints

4- Reframing the Request (Execution vs Product Thinking)

  • Shifted from “produce a file”s
  • To “eliminate the need to manually produce it”

Turned a task into a capability

Implementation

  • Built a client-side crawling tool (HTML + JS) using an AI Agent for fast development
  • Automated directory traversal and PDF detection
  • Structured extracted data into tabular format
  • Enabled one-click export to .xlsx
  • Added real-time logging to increase user trust during execution

Validation

  • Task reduced from 2–3 days to under 5 minutes
  • Successfully used by non-technical staff without guidance
  • Eliminated need for repeated manual effort

Learnings

  • Reframing the problem is often higher leverage than optimizing execution
  • In constrained environments, scope definition is the primary design tool
  • Not every solution needs to scale technically — some need to scale operationally
  • AI-assisted development can accelerate execution without reducing product ownership

Role

Owned the solution end-to-end:

  • Problem framing
  • Strategy and decision-making
  • UX definition
  • Implementation direction (AI-assisted development)

Delivered from initial request to usable product within a single working session.
