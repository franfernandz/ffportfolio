Rebuilding a Legacy Reporting System into a Scalable Data Platform

Replaced a file-based Excel workflow used by 22,000+ users with a real-time web platform, eliminating data duplication and enabling scalable analytics in a legally critical system.

Impact:

  • -100% reduction in file generation
  • +25% faster data entry
  • -12% errors
  • -35% support requests

Project Overview

Led the end-to-end transformation of a legacy Excel-based reporting system into a web-based data platform used by 22,000+ users, eliminating file dependency and enabling scalable data analysis in a legally and financially critical workflow.

Problem

The existing system relied on monthly Excel submissions, generating millions of files over time.

As data analysis needs increased, a workaround was introduced:

  • Extract data from Excel files
  • Store it in a separate database for BI processing

This created a structural inefficiency:

  • Data duplication across systems
  • Increased processing time
  • Higher risk of inconsistencies and errors
  • No scalable way to analyze data in real time

The system was optimized for data submission, not for data consumption.

Insight

The core issue was not the Excel interface itself, but the file-based architecture.

Any solution that preserved files as the source of truth would:

  • Maintain duplication
  • Limit scalability
  • Increase operational complexity

The problem required eliminating the file layer entirely.

Strategy

Replace the file-based workflow with a transactional, web-based system where:

  • Data is created, validated, and stored in real time
  • The database becomes the single source of truth
  • File generation becomes optional (e.g. PDF exports)

At the same time:

  • Preserve familiarity to reduce adoption risk
  • Minimize friction in a mandatory-use system with 22,000 users

Key Decisions & Trade-offs

1- Preserve Excel Structure (Adoption vs Optimization)

Replicated the original form structure instead of redesigning it.

  • Reduced cognitive load and resistance
  • Avoided re-training at scale
  • Limited immediate UX improvements

Chose adoption speed over interface innovation

2 - Web-Based Architecture (Accessibility vs Control)

Selected a browser-based solution over desktop alternatives.

  • Zero installation
  • Immediate updates across all users
  • Lower technical friction
  • Less control over user environment

Maximized accessibility and scalability

3 - Eliminate File Dependency (System Integrity vs Legacy Compatibility)

Removed Excel as the operational unit.

  • Single source of truth
  • Real-time validation and storage
  • No duplication
  • Required redefining internal processes

Prioritized data integrity and long-term scalability.

4- Integrated Data Visualization (Efficiency vs Flexibility)

Enabled in-app data visualization instead of exporting files.

  • Faster access to insights
  • Reduced operational steps
  • Less flexibility for advanced offline use

Optimized for speed and usability in common workflows.

Implementation

  • Built a multi-step web form with validation and progress tracking
  • Implemented real-time data storage
  • Developed internal data visualization tools
  • Introduced confirmation layers for critical actions

Validation

Pilot results (50 users):

  • -100% file generation
  • +25% faster data entry
  • -12% submission errors
  • +20% faster data processing for analytics
  • -35% support requests

Learnings

  • In mandatory-use systems, adoption depends on trust, not choice
  • Familiarity can be a strategic tool, not a design limitation
  • Solving upstream architectural problems has multiplicative downstream impact
  • Small friction in post-submission flows can persist unnoticed without active measurement

Role

Owned the product end-to-end:

  • Problem definition
  • Solution strategy
  • UX/UI design
  • Front-end development
  • Stakeholder alignment

Collaborated with a database specialist for data infrastructure.
